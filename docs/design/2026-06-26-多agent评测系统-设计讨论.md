## 0. 北极星 —— 为什么做这个(比买票更深的意义)

这个系统的**真正目的不是辅助买票**,而是用一套会自我评测的预测系统,**实证地理解"三对问题"**:

1. **v1 独立推理 vs 市场共识** —— 从积分/出线诉求/动机直接推理,能不能赢过市场?
2. **v2(市场+有据微调) vs 纯市场** —— "照抄市场再打补丁"到底有没有加价值?(v2 的存在生死考)
3. **v1 vs v2** —— 独立推理 与 贴市场 两种哲学,各自的主场在哪?

买票只是这套理解的副产品。系统的灵魂是 **Brier 三方对照 + 按场型分桶**,用来回答"**什么场合该信市场/v2,什么场合该信独立推理(v1)**"。

## 1. 现状架构(as-is,均为代码事实,非设想)

**三脑 + 情报 + 看板**:

- **v1** `football-match-predictor`(比分 + 胜平负 + 出线形势,3层 先验/盘口/首发微调)
- **v2** `wc-forecaster-v2`(market-anchored 概率:默认照抄市场基线,仅在有据时偏离;**不出比分当结论、不算价值**)
- **价值** `odds-value-analyst`(竞彩欧赔 × Poly去水 → +EV/最不亏腿,green/yellow/red/skip)
- **deepsearch**(`/deep-research` 多代理工作流 → markdown 报告)

**数据流**: `报告 → 脑 → join → POST → 看板`(单向,看板被动)

- v1、价值 **直接 Read `reports/deep-search-*.md` 原文**(agent 提示词层消费)。
- v2 **不读 md**,只读 `data/wc.db` 的 enrich 表(经 `tools/save_intel.py` 把报告抽取后的 ✅确证事实写入),由 `match_fact_card()` 喂来。
- 盘口: 竞彩本地直连 `odds_watch.py`(launchd 常驻)→ `.cache/odds_cache.db`;欧盘 consensus;**Poly 原走 HK 往返,现正改本地 7897 代理直抓(另一 session 在统一)**。
- 看板在 HK(`http://18.166.71.60:8000`),`POST /api/ingest/predictions` 按 match_key upsert;前端读 `GET /api/decisions`。
- 跑分卡 `tools/v2_report.py` → `reports/预测v2.md`(三方平均 Brier)。

**关键现状(影响后续)**:

- 🔴 **跑分卡现在 n=0**:`match_results` 表为空,`record_result()` **生产代码零调用**,赛果回填靠手动一行命令 → 三盘口全是「配对场数: 0」。**v1/v2/市场谁更准,目前根本没在量。**
- 🔴 **无任何 hooks**(仓库内无 `.claude/settings.json`/`hooks/`,用户级也无),`跑今天` 编排 **纯手动**,报告更新不会自动重算。
- 🔴 **v1⊥v2 红线**:派 v2 时上下文**只准**含 baseline + factcard,**零** v1 输出——护三方 Brier 对照的有效性。
- `match_key` = 竞彩 `zucai_num`(形如 `周四055`),odds_cache 与 wc.db 同键。

## 2. 提议演进(to-be)—— 事件驱动多Agent

把"每天手动跑今天"升级为事件驱动:

- **触发分型,不是"一动全唤醒"**:
  - 价值脑 ← 赔率异动(它本来就吃盘口,最顺)
  - v1/v2 ← 大异动(线跳水=市场吸收新信息)或定时,带阈值 + 去抖
  - **deepsearch ← 时间/新闻触发,不被赔率触发**(赔率动 ≠ 底层事实变;一跑 ~60万token/~10min,太贵)
- **调度器焊死红线**:派 v2 永不含 v1 输出(结构性保证,非靠"记得")。
- **幂等**:看板 upsert 语义,重推安全;需并发锁防重复烧 token。

### 架构形态(暂定倾向)

> **Mac = 脑服务器**(事件调度 + 5 agent,本地 CC 订阅跑模型)
> **HK = 展示/状态服务器**(收 POST、伺服看板)

- 这套**不需服务化、不需换厂商**:模型留 Mac 吃订阅(效果零迁移、Brier 连续性不断),HK 只当便宜的 always-on 显示器+数据库。
- **数据推 HK 已解决**:`跑今天` 第5步本就 POST 到 HK 自己的机器(直连可达,非 github 那种要绕代理)。
- **唯一代价**:吊在"Mac 醒着"上(用户已接受;赛前才密集跑,不需凌晨自醒)。

## 3. 关键设计决策(开放 · 含立场与暂定倾向)

### D1. 模型放哪(本地订阅 vs HK API 直连)
- 服务化唯一买到"脱离 Mac";但 **同模型(Anthropic API + 同 model id)效果中性、却要用 Agent SDK 重搭 CC 这套壳**(skills/Workflow/WebSearch/扇出);**换厂商(GPT/Gemini) = 校准断点,毁掉 Brier 历史连续性**,且 deepsearch 对模型质量敏感。
- **暂定倾向**:Mac脑 + HK看板(不服务化),接受 Mac 依赖;混合可选(重而低频的 deepsearch 留本地,轻而高频的"赔率→价值脑→POST"小环可后置)。
- **待决**:是否/何时服务化;若服务化坚持 Anthropic 同模型。

### D2. v1 去留(核心议题)
- **暂不杀**。理由:① v1 是**比分的唯一来源**(v2 拒出比分);② v1 是 v2 的**对照组**,杀它=弄瞎 v2 的立身之本;③ v1 的价值假设集中在**动机畸形场**(末轮已出线大轮换/默契平/生死战),那正是市场也易错定价处。
- **怎么判**:交给"**按场型分桶 Brier**"(见 §4),而不是现在拍脑袋。要砍也是**先砍概率臂、保比分臂**。
- **多 agent 化不逼你砍 v1**,反而让红线更好守、能"只给畸形场点 v1"。

### D3. 数据到 HK
- **已解决**:本地脑 + POST 到 HK,不需服务化(见 §2)。

## 4. v1 按场型分桶验证 → 落到跑分卡(设计)

### 阻塞 0(总开关):赛果回填闭环
`match_results` 空、`record_result` 无生产调用 → 跑分卡 n=0。**不先把赛果灌进去,任何分桶都是空表。**
解法 = 上轮 Q3 的"赛果驱动回填":比赛结束 ~2h → 抓终比分 → `record_result()` → 跑 `v2_report.py`。**这是分桶上线的前置依赖。**

### 分桶层 1(零改库,纯展示)
复用 v2 已落库的标签当"场型":
- `scenarios`(取自 `backend/scenarios.py` 5 套路,其中 **死亡橡皮擦轮换/默契平/生死战必有胜负** = 末轮动机畸形语义)
- `reliability`(稳/中/乱,`乱`≈动机倒挂)
- 改动集中在 `tools/v2_report.py` 两处:`collect()` 给每场派生 `bucket`;`render()` 按 bucket 切片,各调一遍现成 `aggregate()` 出"常规场/动机畸形场"两套子表。

示意输出:
```
## 胜平负 (had)   配对场数: 18
                v1 Brier   v2 Brier   市场 Brier
全局            0.61       0.55       0.57
├ 常规场(n=12)  0.64       0.53       0.52   ← v1 输给市场
└ 动机畸形(n=6) 0.52       0.58       0.61   ← v1 在这反超
```
→ 这张表就是"v1 该不该留 / 该不该选择性出动"的判决书。

### 注意 & 升级路径(层 2,可选)
- bucket 借的是 **v2 的判断**(post-hoc 不破红线,但 v2 判错场型会一起分错桶,有轻微污染)。
- 更干净 = 给 `data/wc.db` `matches` 加 `matchday` 列(现**完全无**轮次/小组字段),采集脚本回填,按**中立的"末轮 vs 非末轮"**分桶。唯一需动 schema 的点,第一版可不做。

### 硬约束
1. **样本量**:末轮畸形场稀少(每组才 2 场末轮),分桶后单桶个位数 → **只能定性,别拿 5 场判 v1 死刑**;真信号要到赛事深处甚至跨届。
2. **跑分卡只量 v1 的"概率臂"**(只在 had 进 Brier);**v1 的比分臂目前完全没被任何指标量** → 若以"比分"为留 v1 的理由,**必须另立比分指标**(精确命中率/比分距离)。这是当前实现的真空。

## 5. Brier 的本质(已达成的共识理解)

- 量的是**概率校准(贴不贴近真实结果)**,不是"猜没猜中赢家"。公式:Σ(你的概率 − 实际)²,实际发生记1、否则0,**越低越准**。
- 三方 = **v1 独立** / **v2 市场+微调** / **市场共识(尤其 Poly去水=聪明钱)**。v2 夹在 v1 与市场中间。
- 三对问题(§0)= 找"每种方法各自的主场";分桶 = 把这个主场显化出来。

## 6. 其他自动协调流程(均已认可,按价值排序)

1. 🌟 **赛前~1h 首发微调窗 cron**(按场 KO−60min:首发+终盘+重跑受影响脑+POST)—— 最高边际、便宜、纯定时,**比赔率触发更该先做**。
2. **新闻/情报触发的定向 deepsearch**(watchlist 队命中 伤/停/首发关键词 → 单场刷新+save_intel+重跑 v2)。
3. **赛果回填闭环**(= §4 阻塞0,同时也是评测总开关)。
4. **按异动大小分级升级**(小波动→只刷价值脑;中→+v1/v2;大+新闻→+deepsearch)。
5. **每场倒计时状态机**(T-24h 全量 deepsearch → T-3h 刷新 → T-1h 首发窗 → 赛后回填)。
6. **背离告警(human-in-the-loop)**:v1/v2/市场劈叉大,或 Poly与竞彩背离超阈值 → 推通知,不自动下结论。
