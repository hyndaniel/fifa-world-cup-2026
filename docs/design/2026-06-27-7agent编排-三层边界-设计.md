## 0. 北极星(承接昨天,不变)

系统的真正目的**不是辅助买票**,而是**实证地理解"模型推理能力 vs 市场情绪/聪明钱判断的偏差"**——即昨天 memo §0 的三对问题:

1. v1 独立推理 vs 市场共识
2. v2(市场+有据微调)vs 纯市场
3. v1 vs v2

买票是副产品。系统的灵魂仍是 **Brier 三方对照 + 按场型分桶跑分卡**。
🔴 **头号阻塞依旧未解**:`match_results` 空、`record_result()` 零生产调用 → 跑分卡 n=0 → "谁更准"目前根本没在量。任何华丽功能都不能挤掉这条闭环。

## 2. 骨架四要素(本轮拍定)

### 决策 1 — 边界切法:按"描述 / 决策 / 调度"三层切(不按数据源切)

| 层 | Agent | 职责 | 不做 |
|---|---|---|---|
| 描述 | **赔率分析 agent**(裂自 odds-value-analyst) | 取亚/欧/竞彩/Poly 共识、去水、市场共识赛果、**异动检测** | 不下价值判断 |
| 决策 | **下注 agent**(裂自 odds-value-analyst) | 综合 v1/v2/共识/价值 → 选腿、评用户方案、讲结算机制;现有 +EV/最不亏归此层 | 不进跑分卡评测 |
| 调度 | **编排层(吸收"分析6")** | 检测异动/新闻 → 决定唤醒谁、跑不跑 deepsearch | **绝不产出喂脑的"判断内容"** |

> 关键:用户原拆的"分析 agent(6)"不是内容分析师,而是**触发路由器**,并入编排层。否则它产出"建设性意见"喂 v1/v2 = 开红线后门。

**唯一要动的现有代码**:`odds-value-analyst` 一分为二——"取共识/去水"半下沉为赔率分析 agent,"+EV/选腿"半上浮为下注 agent。其余三脑边界不变。

### 决策 2 — 编排形态:混合(Workflow 焊红线 + 薄 LLM 判模糊点)

- **骨架 = 确定性 Workflow 脚本**(JS,`pipeline`/`parallel`/`phase`):控制流写死,"派 v2 时上下文零 v1 输出"是**代码结构保证**,不靠任何 agent"记得"。`跑今天` 是其雏形。
- **薄 LLM 层**只碰规则化不了的灰区判断:"这条新闻算不算重大到要触发 deepsearch""异动算大算小"。输入被严格限定,不得引入跨脑污染。
- 拒绝纯 LLM 编排 agent:红线靠 LLM 自觉 = 迟早破。

### 决策 3 — 数据流红线(守边界的物理实现)

"agent 不越界"= 一张画死的数据流图。`→` 表示"喂":

```
                    ┌──────────────┐
   报告 agent ──────▶│ 经 save_intel │──▶ ✅确证事实
  (deepsearch)       └──────────────┘     │
       │                                  ├──▶ v1 预测脑 ─┐
       │  (原文 md)                        │              │
       └──────────────────────────────────┤              │
                                           │              ▼
   赔率分析 agent ─── 共识/去水/盘口 ───────┼──▶ v2 概率脑 ─┼──▶ 下注 agent ──▶ 看板
   (描述市场)         │                    │  (market       │  (决策:选腿/
       │             │                     │   anchored)    │   评方案)
       │             └── 异动信号 ──────────┘                │
       │                      │                             │
       │                      ▼                             │
       └──────────────▶ 编排层(吸收分析6=路由器)◀─新闻信号─┘
                        触发谁、跑不跑 deepsearch
```

**允许的边**(现状已有 / 合理):
- 报告 → v1(读原文 md)、→ v2(经 `save_intel` 只进 ✅确证事实)、→ 下注
- 赔率分析 → v1(其盘口微调层本就吃盘口)、→ v2(market baseline)、→ 下注
- v1 → 下注、v2 → 下注、共识 → 下注(下注是终点,综合谁都行)

**焊死的红线(物理禁止的边)**:
- 🔴 **v1 ↔ v2 双向断开**(本轮升级):派 v2 时零 v1 输出 **且** 派 v1 时零 v2 输出。两条哲学彻底独立,三方 Brier 对照最干净。
  - 注:不影响 v1 的盘口微调层——它吃市场**盘口**,不是 v2 的**输出**,两回事。
- 🔴 **编排层(分析6)只发"触发信号",绝不发"判断内容"喂脑**。路由器,不是分析师。

### 决策 4 — 下注 agent 不进评测

下注 agent 在终点把 v1+v2+共识+价值揉一起做决策——这**不破** Brier 红线,因为:
- 跑分卡量的是 v1 / v2 / 市场**各自独立落库的预测值**;
- 下注 agent 的"综合判断"是给用户买票用的**下游产物**,不落预测库、不参与三方对照。

## 3. 落地排序(我的建议 · 本轮未承诺,待续)

用昨天的头号阻塞定第一锤:

1. **🥇 赛果回填闭环**(地基/总开关)——赛后 ~2h 抓终比分 → `record_result()` → 重跑 `v2_report.py`。**点亮整个北极星,最便宜(纯定时+抓取,不烧脑)。**
2. **🥈 编排骨架成形**——`跑今天` 升级成正式 Workflow:三层边界落位、红线写进脚本结构、赔率分析/下注从 `odds-value-analyst` 裂出。
3. **🥉 事件驱动 + 首发窗 cron**——赛前 KO−60min 首发微调窗(memo §6 最高边际)+ 异动/新闻触发的薄 LLM 路由。
4. 报告回灌、购彩记录统计、背离告警等增量功能,挂骨架上逐个长。

核心主张:**先点亮评测闭环(证明系统有意义)→ 再搭编排骨架(让它可复用)→ 最后才叠华丽功能。** 反序 = 昨天的坑(n=0 还在)上再埋一层。
