## 2. 骨架四要素(已拍定)

### 决策 1 — 边界切法:按"描述 / 决策 / 调度"三层(不按数据源切)

| 层 | Agent | 职责 | 不做 |
|---|---|---|---|
| 描述 | `wc-odds` | 取亚/欧/竞彩/Poly 共识、去水、市场共识赛果、**异动检测** | 不下价值判断 |
| 决策 | `wc-bet` | 综合 v1/v2/共识/价值 → 选腿、评用户方案、讲结算;现有 +EV/最不亏归此 | 不进跑分卡评测 |
| 调度 | `wc-orchestrator`(吸收"分析6") | 检测异动/新闻 → 决定唤醒谁、跑不跑 deepsearch | **绝不产出喂脑的"判断内容"** |

### 决策 2 — 编排形态:混合(Workflow 焊红线 + 薄 LLM 判模糊点)

- **骨架 = 确定性 Workflow 脚本**(JS,`pipeline`/`parallel`/`phase`):控制流写死,"派 v2 时上下文零 v1 输出"是**代码结构保证**,不靠 agent"记得"。`跑今天` 是雏形。
- **薄 LLM 层**只碰规则化不了的灰区:"这条新闻算不算重大到触发 deepsearch""异动算大算小"。输入严格限定,不引入跨脑污染。
- 拒绝纯 LLM 编排 agent:红线靠自觉 = 迟早破。

### 决策 3 — 数据流红线(守边界的物理实现)

`→` 表示"喂":

```
                    ┌──────────────┐
   wc-intel ───────▶│ 经 save_intel │──▶ ✅确证事实
  (deepsearch)       └──────────────┘     │
       │                                  ├──▶ wc-score-v1 ─┐
       │  (原文 md)                        │                │
       └──────────────────────────────────┤                │
                                           │                ▼
   wc-odds ───── 共识/去水/盘口 ───────────┼──▶ wc-prob-v2 ─┼──▶ wc-bet ──▶ 看板
   (描述市场)        │                     │  (anchored)    │   (决策)
       │            │                      │                │
       │            └── 异动信号 ──────────┘                │
       │                     │                              │
       │                     ▼                  赛后  ┌─────┴─── wc-review
       └──────────▶ wc-orchestrator(路由器)◀─新闻信号 │      (定性归因·见§4)
                     触发谁、跑不跑 deepsearch          └─ record_result → 跑分卡
```

**允许的边**(现状已有 / 合理):
- wc-intel → score-v1(读原文 md)、→ prob-v2(经 `save_intel` 只进 ✅确证事实)、→ wc-bet
- wc-odds → score-v1(盘口微调层本就吃盘口)、→ prob-v2(market baseline)、→ wc-bet
- score-v1 → wc-bet、prob-v2 → wc-bet、共识 → wc-bet(下注是终点,综合谁都行)
- **wc-review ← 一切(但都在赛后)**:读冻结的三方预测 + 实际 + 当时情报 + 跑分卡数字

**焊死的红线(物理禁止的边)**:
- 🔴 **score-v1 ↔ prob-v2 双向断开**:派 v2 时零 v1 输出 **且** 派 v1 时零 v2 输出。两哲学彻底独立,三方 Brier 对照最干净。(不影响 v1 盘口微调层——它吃市场**盘口**,非 v2 的**输出**。)
- 🔴 **wc-orchestrator 只发"触发信号",绝不发"判断内容"喂脑**。路由器,不是分析师。
- 🟢 **wc-review 红线安全**:它看全三方,但**全发生在赛后**——预测时刻的 v1⊥v2 隔离未被破坏(同 wc-bet、跑分卡)。

### 决策 4 — 下注与复盘不进评测

`wc-bet`(向前决策)和 `wc-review`(向后归因)都综合三方,但**不破** Brier 红线:跑分卡只量 score-v1 / prob-v2 / 市场**各自独立落库的预测值**;bet/review 是下游产物,不落预测库、不参与三方对照。

## 3. 命名与产出规范(本轮新增)

核心思想:**agent-id 直接前缀它的报告,谁产的一眼可见。** 起因:现有报告名在骗人——`预测v2.md` 听着像 v2 的预测,**实为三方跑分卡**(系统级,非 v2 独有);`小组赛比分预测.md`/`盘口下注复盘.md` 名里完全没有产出者。

### 3.1 报告布局

```
reports/
  agents/    wc-score-v1__比分预测.md   ← 台账,追加(原 小组赛比分预测.md)
             wc-bet__下注复盘.md         ← 台账,追加(原 盘口下注复盘.md)
             wc-review__复盘归因.md      ← 台账,追加(新)
  scoring/   三方跑分卡.md               ← 台账(正名自 预测v2.md,它不是v2的)
  intel/     2026-06-26__赛前情报.md     ← 快照,日期版,冻结(原 deep-search-*.md)
  _archive/  跨模型实验等死文件          ← 非 agent 产出(model-comparison / fable / gpt)
  _state/    *.json 机器态(盲测快照等)  ← 数据/状态,非给人看的报告
```

### 3.2 台账 vs 快照(单文件追加 vs 日期版本)—— 看报告"性质",不一刀切

| 性质 | 例子 | 怎么存 | 为什么 |
|---|---|---|---|
| **台账**(累积、要回填实际) | 比分预测、下注复盘、复盘归因、三方跑分卡 | **长青单文件,持续追加** | 价值在"跨天连续序列";命中率/Brier 是累积算的,拆日文件就打碎;回填实际必须和当初预测同行 |
| **快照**(某刻冻结、事后不可改) | 赛前情报、盲测快照 | **日期版本,一天一文件,写完冻结** | 价值恰是"赛前那刻所知";事后追加/改写就毁真盲测 |

> 一句话原则:**要算累积指标/要回填实际的 → 追加;要保住"当时所知"的 → 冻结日期版。**
> 世界杯 ~64 场/一月,台账单文件几十条目轻松。

### 3.3 历史迁移(确认:改名搬走)

- 台账类(比分/复盘)本就单文件,直接重命名。
- 🔴 **情报快照搬时只改名、绝不合并**:`deep-search-*-06-25.md`、`-06-26.md` 各是冻结记录,并成一个就毁盲测。
- `_archive/` 的跨模型实验、`_state/` 的 .json 不对应任何 agent,挪出 agent 语义空间(下划线=非 live)。

### 3.4 agent.md 统一骨架(点3:复写模板)

现三个 agent.md 画风/长度不一(64/84/36 行,字段不齐)。统一成五段固定模板:
1. **身份**(一句话:我是谁、什么脑)
2. **边界**(do / **don't** + 越界问题路由给谁)
3. **输入来源**(读哪些缓存/库/报告)
4. **输出落点**(写哪个**规范报告名**,台账还是快照)
5. **红线**(本 agent 头上的硬约束)

三个现有的按此重提炼、长度对齐;新 agent 同模板新建。

### 3.5 迁移现实约束(所以本轮只写 spec、不执行)

- 改名是**横切重构**:`reports/小组赛比分预测.md` 等硬编码散在 agent.md、`save_intel.py`、`v2_report.py`、`跑今天` skill,得逐处跟改,需专列迁移清单。
- 改 `.claude/agents/*.md` **撞自修改护栏**,需用户在自己消息里明确授权,绝不 Bash 绕过 → 复写是后续单独授权步骤。

## 4. wc-review(复盘归因)—— 两段式,严护栏

"复盘"实为三件性质不同的事,分别归位:

| 子任务 | 性质 | 归属 |
|---|---|---|
| ① 回填实际比分 | 机械(抓终比分→`record_result`) | `wc-orchestrator`(回填闭环),非 agent |
| ② 定量裁决:谁更准 | 确定性计算(三方 Brier/分桶) | 跑分卡 tooling,非 LLM |
| ③ 定性归因:方向对/错 + 为什么 | LLM 叙事判断 | **`wc-review` agent** |

**`wc-review` 的铁律护栏**:末轮畸形场样本个位数,在 n=3 上写"为什么错"= "拿 5 场判 v1 死刑"的散文版,比数字版更诱人更不严谨。故:
- **只产出"假设 + 记录",绝不下结论;裁判永远是定量跑分卡。** 归因是"喂分桶卡去统计验证的假设",不是判决。
- **排在最后**:②跑分卡 n=0 没解前,③没数字可锚,纯讲故事。_(2026-06-29 注:n=0 已解、n=31;但 wc-review 仍"样本够再上",排序结论不变。)_

**落地两段式**:
- **先做"机械对错标"**(随①回填闭环):回填时给复盘台账盖纯事实标签 `v1✅/v2❌/市场✅`,无 LLM、零讲故事风险。
- **后做 `wc-review` agent**(③):样本够了再上叙事归因,永远带"仅假设、不下结论"帽子。

## 7. 项目编码规范(本轮新增 · 轻量,只钉真在用的,不堆 lint)

1. **字符与文件**:源码/报告/注释一律 **UTF-8**;中文用于报告正文、注释、commit scope 完全 OK;文件名优先 ascii,报告名用 §3 的 `wc-*__中文.md` 混排可。`git config core.quotepath false` 让中文文件名在 git 输出里可读。
2. **Python**:**优先 stdlib**(承接 Poly/采集脚本 pure-stdlib 习惯),引第三方库要有理由;每脚本可单独 `python3 xx.py` 跑调试,失败 **exit 非零 + 打印错误**(承接现有 error contract)。
3. **提交信息**:沿用 `type(中文scope): 说明`,type ∈ feat/fix/docs/style/chore/refactor;PR 标题同;结尾带 `Co-Authored-By`。
4. **命名**:agent/报告沿用 §1 `wc-<角色>-<版本>` 家族 + §3 agent-id 前缀报告。
5. **红线常驻**:见下方"约束速查"——本地登录、score-v1⊥prob-v2 双向、改 agents/ 撞护栏、ET 铁律、看板密码不入库、不碰资金。
6. **不入库**:运行时态(`*.db`、`.cache/`、`.claude/agent-memory/`、`docs/superpowers/handoffs/`、`config.toml`)一律 gitignore;机器态 json 进 `reports/_state/`。
