# 架构文档 — WC 预测 × 价值看板

> 本文讲**为什么是这个形状**、**各块怎么连**、**有哪些不可妥协的不变量**。
> "是什么、怎么跑、怎么部署"看 [`README.md`](../README.md);单个功能的设计细节看 `docs/superpowers/specs/`。
> 受众:未来的自己 + 接手的人。读完应能回答:一条预测从情报到手机看板,中间经过谁、谁不能看见谁、什么时候会过时。

---

## 0. 一分钟心智模型

系统里有**四个角色**,只有三个是"脑":

| 角色 | 是不是脑 | 一句话 |
|---|---|---|
| **deepsearch 情报层** | ❌ 是**粮草**,不下结论 | 多智能体跑出中立赛前情报(伤停/出线形势/历史),不预测、不估价值 |
| **v1 比分脑** | ✅ | 从出线形势+先验推**比分**,自评胜平负% |
| **v2 概率脑** | ✅ | **抄市场概率**当基线,只在有硬证据时偏离 |
| **价值脑** | ✅ | 竞彩赔率 × 聪明钱去水概率 → **价值/最不亏的腿** |

三个脑**互相隔离**地吃同一份粮草,各产一份结论,最后 join 成一张**决策卡**推到手机看板。
隔离是刻意的:独立的脑**撞车=可信,打架=警报**,而 v1⊥v2 的隔离还是三方跑分有效性的前提(见 §3.1)。

### 1.2 运行拓扑(物理上跑在哪)——三块割裂,各有原因

```
   ┌────────────── 本地 macOS(订阅登录态)─────────────┐        ┌──────── AWS-HK 机房 ────────┐
   │                                                  │        │                              │
   │  Claude Code(三脑推理 + 一键编排)                 │  POST  │  FastAPI backend/web.py      │
   │  竞彩直连(住宅 IP 不被 WAF 拦)                    │ ─────▶ │  data/wc.db(决策卡库)        │
   │  .cache/odds_cache.db(本地, 不同步)              │  决策卡 │  前端静态页(决策/看板/报告)  │
   │                                                  │        │  Polymarket 直连可达          │
   │        │ remote-agent MCP                        │        │  足彩须走住宅 SOCKS5 代理     │
   │        ▼                                          │        └──────────────────────────────┘
   │   aws-hk 抓 Polymarket ──远程exec/download──▶ 回灌本地缓存                                  │
   └──────────────────────────────────────────────────┘
```

**为什么割裂(三条物理约束,见 §6):**
1. **订阅制无 API key** → LLM 推理只能在本地 macOS 登录态跑,HK 后端**不能直调模型**。
2. **GFW** → 本地直连 Polymarket 不通。无人值守的 `refresh_all`(launchd)现经**本地 Clash 7897 代理**直抓;交互式「跑今天」仍走 remote-agent(HK)抓回回灌(两条路径并存)。
3. **EdgeOne WAF 按 IP 拦** → HK 数据中心 IP 抓不到足彩,须走出口为住宅 IP 的 SOCKS5 代理。

### 1.3 赛后回填闭环(承重墙:让"谁更准"可量)

预测落盘只是上半场;**赛后自动回填终比分**才让三方跑分卡有数据可算:

```
  sporttery 开奖端点(getUniformMatchResultV1.qry,key=zucai_num 与预测天然同键)
        │  tools/backfill_results.py --once(launchd com.wc.backfill-results 每5min 幂等重跑)
        ▼
  record_result() → .cache/odds_cache.db · match_results 表
        ├──▶ 重生成 reports/复盘对错标.md(机械对错标:三方 had argmax ✅中/❌错/—无预测,纯事实无 LLM)
        └──▶ 重跑 tools/v2_report.py → reports/预测v2.md(三方 Brier 跑分卡 + 场型/轮次分桶)
```

幂等 + 从 db 全量确定性重生成 = 每 5min 无害重跑、上次崩下次自愈;抓取**纯 stdlib(urllib)无第三方依赖**(可用系统 /usr/bin/python3 跑)。
这条闭环是整个北极星(实证「v1 独立推理 vs v2 市场锚定 vs 纯市场」谁更准)的**总开关**——没它跑分卡 n=0、"谁更准"根本没在量。

## 2. 四个角色详解

> Agent 定义在 `.claude/agents/*.md`。改这些文件会撞自修改护栏,须用户显式授权。

### 2.1 deepsearch 情报层(粮草,非脑)

- **产出**:`/deep-research` 多智能体工作流 → `reports/deep-search-*.md`,中立赛前情报(出线形势 / 伤停停赛 / 历史交锋 / 近期状态 / 动机)。**不含比分、胜平负、盘口价值结论。**
- **下游分发**(同一份报告,三方各取所需):
  - **v1 / 价值脑**:直接读 prose 原文(它们靠叙事吃饭)。
  - **v2**:**不读原文**。经 `tools/save_intel.py` 抽取**确证事实**(停赛/伤停/官宣轮换)→ 写 `data/wc.db` enrich 表 → v2 经 `match_fact_card()` 只看到一张干净的事实卡。
- **置信分层**:"官宣某主力停赛" = 硬,高置信;"这队应该想保平" = 叙事,软。只有硬事实进 v2 口(见 §3.2)。

### 2.2 v1 比分脑 — `wc-score-v1`

| | |
|---|---|
| **输入** | 实力/状态/历史报告、三源盘口、首发名单、出线诉求、deep-search **原文** |
| **方法** | 出线形势(积分→净胜球→进球→相互战绩)+ 三层叠加(先验 / 盘口 / 赛前~1h 首发微调) |
| **输出** | 比分(主-客) + 自评胜平负% + 出线诉求 + 一句依据 |
| **不做** | ❌ 不算价值/EV、不判陷阱盘 → 交给价值脑 |
| **跑分** | **命中率**,落 `reports/小组赛比分预测.md`(赛前落盘、赛后回灌) |

### 2.3 v2 概率脑 — `wc-prob-v2`

| | |
|---|---|
| **输入** | **仅两样**:① 市场去水基线 `baseline_market()`(胜平负/让球/总进球)② 中立事实卡 `match_fact_card()` |
| **方法** | 逐盘口抄基线 → **默认照抄** → 仅当有非陈旧、相关的确证事实时才偏离 → 每条偏离带 `factor_source` 短引用 → 打靠谱度(稳/中/乱)+ 剧本标签 |
| **输出** | 逐盘口概率 + 靠谱度 + 剧本标签 + 是否偏离 |
| **不做** | ❌ 不出精确比分、不算 EV、不推下注、不读 v1、不读出线形势 |
| **跑分** | **Brier 分**(校准度硬指标),`tools/v2_report.py` → `reports/预测v2.md` |

### 2.4 盘口 — `wc-odds`(描述)/ `wc-bet`(决策)

单体价值脑已按「描述 / 决策」三层边界裂成两个 agent(见 `docs/design/2026-06-27-7agent编排-三层边界-设计.md` §2)。

**`wc-odds`(描述层)**

| | |
|---|---|
| **输入** | ① 竞彩欧赔(`source=zucai`)② Polymarket 去水(`source=poly`,**须今日**)③ 欧盘共识(`source=consensus`)④ deep-search 原文(陷阱盘/动机段) |
| **方法** | 去水算隐含概率;报市场共识、竞彩vs共识分歧、盘口异动 |
| **输出** | 去水/共识/分歧/异动 + 陷阱盘/结算结构客观描述 |
| **不做** | ❌ 不下价值判断/不选腿 → 交给 wc-bet;❌ 不预测比分 → 交给 v1 |

**`wc-bet`(决策层)**

| | |
|---|---|
| **输入** | wc-odds 的去水/共识/异动 + v1/v2 预测 |
| **方法** | `value = 竞彩欧赔 × Poly去水概率` → 分档(🟢🟡🔴⚪)→ 综合三方选腿 |
| **输出** | 价值表 + 最不亏的腿 + 评用户方案 + 结算/陷阱规避 + A价值单关/B清醒彩票 |
| **不做** | ❌ 不自取盘口(找 wc-odds)❌ 不预测比分(找 v1)❌ **不进 Brier 跑分卡** |
| **跑分** | 落 `reports/盘口下注复盘.md`;**置信度完全取决于 Poly 新鲜度**(见 §3.3) |

## 3. 核心不变量(红线,改动前先读)

### 3.1 🔴 v1 ⊥ v2 防火墙(不可妥协)

派 v2 的 prompt 里**绝不能出现任何 v1 的比分/概率/推理**。实现:三脑各为一次独立 subagent 调用(共 3 次),其中**红线特指 v1 与 v2 这两次上下文互不可见**(价值脑不进 Brier 对照,无此约束)。

**为什么**:三方 Brier 跑分(v1 vs v2 vs 纯市场基线)要有效,三者必须独立。v2 一旦瞥见 v1,两者相关,跑分失真——你就分不清"v2 真比市场强"还是"v2 抄了 v1 的作业"。

### 3.2 v2 只吃确证事实

`save_intel.py` 只搬**停赛/伤停/官宣轮换**这类硬事实进 enrich 表,剥掉出线形势/动机/叙事/盘口赔率。

**为什么**:v2 的命是 market-anchored(锚定市场)。叙事一旦进来,它会被"故事"带着乱偏,失去校准。市场是几十家庄家+聪明钱博弈出的有效价格,本就难赢——v2 的本分是默认信市场,只在有铁证时挪一点。

### 3.3 新鲜度窗口(陈旧 = 误导)

| 数据 | 时间戳/窗口 | 过期后果 |
|---|---|---|
| **盘口**(竞彩/Poly/共识) | `odds_cache.ts` 北京时区,秒精度,取 `ORDER BY ts DESC` 最新 | **陈旧 Poly 伪造"假黄档"**:本该 🔴 的显示成 🟡,你以为不亏其实亏 |
| **intel 事实卡** | `match_fact_card(stale_hours=48)`;`age>48h` 或 `age<0`(源时钟偏差)或不可解析 → `stale=true` | stale 事实**沉底、不驱动 v2 偏离** |
| **WebFetch 同一 URL** | 缓存 **15 min** | 比分页卡旧分钟,4-5min 狂刷只吃旧缓存(用户看直播永远比你快) |

原则:**取不到/缺位就明说哪源缺,不拿陈旧/缺失硬算**;缺则相关结论**降靠谱度并标注**。

## 5. 一键编排:`跑今天`(7 步)

`.claude/skills/跑今天/SKILL.md` —— 说「跑今天」即按序走完(~80% 自动化):

1. **按 ET 当下时刻**判定该预测哪几场(CST=ET+12/13;温哥华 PT=ET−3)
2. **刷三源盘口**:`odds_watch.py --once`(竞彩)· `--consensus`(欧盘)· `poly_fetch_hk.py` 经 remote-agent(Poly)
3. **并行派三条独立 pass**(死守 §3.1 红线)→ 各写回缓存 + reports
4. 跑 **Brier 跑分卡**:`tools/v2_report.py` → `reports/预测v2.md`
5. **跨库 join 成 Decision 对象** → `POST /api/ingest/predictions` 写回 HK 看板
6. 一段话给用户**扫读总结**
7. **诚实边界**:赛前~1h 首发微调窗仍需手动再触发一次

### 5.1 时序:什么时候要重跑?(§5 的延伸)

随时间变的只有两类输入——**行情**和**首发**——所以重跑价值**集中在节点**,不是均匀的:

- **提前几小时**跑第一遍:行情还飘、消息没齐,当初稿。
- **行情明显跳水**时重跑:独赢/让球/大小球短时大幅移动 = 市场吃到新信息,v2 基线该刷。
- 🌟 **最高价值的一次 = 赛前~1h 首发出炉后**:首发与原依据冲突 **且 盘口跟进** → v2 可能产生有据偏离、价值档可能翻档。这是边际收益最高的窗口(第 7 步留的口子)。
- ⚠️ 陷阱:**首发变了但盘口没动 → 多半已 priced-in,不改**。
- 微妙点:**越近开赛市场越有效,v2 能加的料越少**(偏离趋近 0)。临门一脚那跑主要价值是"用最准基线+首发硬事实定稿",别指望 v2 翻盘式新观点。

**实操**:一天两跑足够(开赛前几小时铺底 + 赛前~1h 定稿),中间除非盘口大跳水不必反复跑。

## 6. 运行约束(为什么拓扑这么割裂)

| 约束 | 后果 | 应对 |
|---|---|---|
| **订阅制无 API key** | LLM 只能本地 macOS 登录态跑;HK 后端不能直调模型 | 推理留本地,HK 只做存储/展示;网页触发 LLM 须靠任务队列桥接 |
| **GFW** | 本地直连 Polymarket 不通 | 无人值守 `refresh_all` 经本地 Clash 7897 代理直抓;交互式跑今天经 remote-agent(HK)`remote_exec`→`remote_download` 回灌(两路并存) |
| **EdgeOne WAF 按 IP 拦** | HK 数据中心 IP 抓不到足彩 | 足彩流量走出口为住宅 IP 的 SOCKS5 代理(`[zucai] proxy`) |

## 7. 置信度模型(谁更靠谱)

从高到低:

1. **v2 最校准**——锚定有效市场价格,只在有硬证据时挪。代价是**很少给惊喜**(多数时候≈市场)。唯一用 **Brier 分**长期跟踪。
2. **价值脑**——置信度**完全看 Poly 新鲜度**:今日 Poly→可信;旧→假黄档。结论永远附带"数据多新"的前提。
3. **v1 主观性最高**——靠出线形势/战意/先验推,更易与市场不一致;有时这正是独立第二意见的价值,有时就是错。用**命中率**跟踪。
4. **deepsearch 看"坐没坐实"**——硬事实高、叙事软;只有硬的喂 v2。

> 一句话:**v2 最校准但最"无聊",v1 最有观点但最易错,价值脑成败全看 Poly 新不新,deepsearch 提供原料不提供结论。** 三脑独立 → **撞车=可信,分歧=警报**。

## 8. 跑分 / 追踪(三本独立账)

| 脑 | 指标 | 账本 |
|---|---|---|
| v1 | 命中率 | `reports/小组赛比分预测.md` |
| v2 | **Brier 分**(+三方对照:v1/v2/纯市场基线) | `reports/预测v2.md`(`tools/v2_report.py` 渲染) |
| 价值 | 价值档命中/盈亏复盘 | `reports/盘口下注复盘.md` |

三本分开,因为三脑量纲不同(比分 vs 概率校准 vs 价值期望),不能混算。

> 实际结果由 §1.3 回填闭环(live 作业 `com.wc.backfill-results`)自动驱动:v2 的三方 Brier 跑分卡与机械对错标台账每 5min 从 `match_results` 全量重生成;v1 命中率/价值盈亏的散文复盘仍由对应 agent 赛后回灌。
