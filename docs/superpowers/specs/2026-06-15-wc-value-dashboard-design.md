## 1. 背景与目标

### 1.1 起因
用户在 2026 世界杯期间用中国体彩竞彩（足彩）打串关，连挂 4 张（净亏 200 元）。复盘发现：足彩对每个结果都有 ~13% 抽水，几乎注注 -EV；串关把多个 -EV 相乘，期望塌陷、方差爆炸，注定长期巨亏。同时 Polymarket（聪明钱预测市场，抽水仅 1-2%）能当"真实概率"标尺，对比足彩可识别被高估/低估的盘口。已有原型 `wc_value.py` 验证了"足彩 × Poly 价值对比"思路可行。

### 1.2 目标
做一个**部署在 AWS-HK、手机浏览器可实时访问的双模式看板**：

- **模式 A（价值单关 / 赚小钱）**：实时拉两平台盘口，算价值，只标出真 +EV / 接近公允的**单关**机会，配纪律（小注、记账、不追损）。
- **模式 B（清醒彩票 / 博一把）**：用户想博高赔串关时，用最不亏的腿凑，但**把真实命中率与期望明明白白展示**，当娱乐花、固定预算、不自欺。

### 1.3 诚实定位（非目标）
- **不是**"稳定回血/盈利机器"——足彩长期 -EV，没有工具能改变这一点。
- **不**自动下注、**不**碰用户资金、**不**构成投资建议。
- 价值在于：**把"几乎必亏"改善为"大致打平、偶尔薄赚 + 守住下限 + 不上头"**，并用数据让用户保持清醒。

---

## 2. 核心原则

1. **双钱包物理隔离**：A=价值单关预算，B=博彩票预算。固定额、不互相补、不"回血"加注。
2. **单关优先**：默认推单关，杀掉串关的方差/期望塌陷。串关只在模式 B 显式构造，且强制展示真实概率。
3. **只打价值**：以 Polymarket 概率为真值基准，只推 value≥阈值（足彩赔率 × Poly概率 > 1）的点，其余不碰。
4. **诚实展示**：薄边（value 1.0~1.03）标注"在抽水噪声内、接近公允非铁 +EV"；串关展示真实命中率 + 期望。
5. **纪律内建**：注码上限、预算剩余、停售倒计时、追损提醒由系统强制，不靠自觉。
6. **记账打脸**：A/B 分开记真实盈亏 ROI，用数据照镜子。

## 3. 系统架构

单台 **AWS-HK VPS**。香港出口实测可**同时直连足彩(CN) 与 Polymarket(US)，无需代理**（关键架构前提，已验证）。

```
[AWS-HK · docker-compose]
  ┌─ poller ──────── 每 1–5min 拉 足彩(sporttery) + Polymarket(gamma)，直连
  │                   写盘口快照入 SQLite（含历史，用于分歧/异动）
  ├─ value-engine ── 复用 wc_value 逻辑：足彩×Poly → value/EV，标 🟢/🟡
  ├─ lineup-watcher ─ 开赛前 ~1h：WebSearch 拉首发 + 新闻链接
  ├─ llm-worker ───── Claude API，事件驱动（首发出炉 / 分歧拉大）才调用，控成本
  ├─ web (FastAPI) ── 暴露 /state JSON + serve 静态前端 + /reports（markdown 渲染）
  └─ SQLite ──────── 盘口历史 / 价值 / watchlist / 账本 / 配置
        ↑
   手机/电脑浏览器 ── 单页前端每 N 秒轮询 /state 自动刷新；访问加口令
```

**为什么单机直连**：HK 出口两边都通 → 不用代理（去掉最脆的一环）；常驻不睡眠；浏览器随时访问。

## 4. 数据源

| 源 | 接口 | 内容 | 备注 |
|---|---|---|---|
| 足彩 | `webapi.sporttery.cn/.../getMatchCalculatorV1.qry?poolCode=had,hhad,ttg` | 胜平负 / 让球胜平负 / 总进球 | 需 iPhone UA + `Referer: m.sporttery.cn` |
| Polymarket | `gamma-api.polymarket.com/events?tag_id=102232`（FIFA World Cup）、`?slug=fifwc-<主><客>-日期`、`<slug>-more-markets` | 独赢/让分(spread)/大小球(O/U)/BTTS，价格=概率 | slug 队码 FIFA/ISO 混用 |
| 首发/新闻 | WebSearch（`<team> starting XI lineup`） | 预测/确认 XI + 来源文章 URL | 开赛前 ~1h；LLM 摘要 |

**配对键**：用"对阵"（两队英文名子串）唯一匹配足彩↔Poly，免时区日期烦恼（小组赛两队只碰一次）。CN→EN 队名映射表内置（48 队）。

## 5. 价值计算

`value = 足彩欧赔 × Poly真实概率`（>1 = +EV）。映射规则（已在 `wc_value.py` 实现并验证）：

- **胜平负(had)** → Poly 独赢 主/平/客。
- **让球(hhad)** → Poly 让分(spread)：让-k 主胜=主净胜≥k+1=cover[k+0.5]；让-k 平=净胜恰好 k=cover[k-0.5]-cover[k+0.5]；受让+k 用客队 cover。
- **总进球(ttg) 恰好 N** → O/U 差：over[N-0.5]-over[N+0.5]。
- **假阳性防护**：高分桶（6/7+）若 Poly 缺对应高 O/U 线，概率会被高估成伪 +EV → **自动跳过并标注**（原型已发现并修复此坑）。
- **抽水提示**：Poly 概率含 1-2% vig，value 仅略>1 视作接近公允。

## 7. 特别关注（Watchlist）

可 pin 三种对象：

- **球队**：关注的队，其比赛置顶。
- **场次**：关注的整场对阵。
- **球员**：关注某人——首发确认/缺阵时告警。

pin 项展示：该场价值点 + 首发（出炉后）+ **新闻链接（可点）** + 停售倒计时。**变化即告警**：首发出炉 / 冒出价值点 / 两平台分歧拉大 / 关注球员首发状态确定。

---

## 8. 双模式

### 模式 A — 价值单关
价值雷达只列单关价值点；点开给建议注码（占 A 钱包固定小比例，如 1-2%）。绝不推串。

### 模式 B — 清醒彩票构造器
用户设目标（"博 X 万" 或 "选 N 条腿"）→ 工具用价值表挑**最不亏的腿**凑高赔串 → **强制展示**：总赔率、**真实命中率**、**期望%**、🎟️"这是彩票，默认已花掉"标签。下注计入 B 钱包，扣预算。不让赔率假象骗人。

---

## 9. 记账与风控

- **A/B 钱包独立**：各自预算、已下注、实际结果、累计盈亏/ROI。
- **强制纪律**：单注上限、B 周预算剩余显示、停售前"该下手/放弃"提醒、**追损检测**（短时连续加注 → 弹提醒）。
- **诚实数字**：账本长期大概率为负（足彩本质），系统如实展示，不美化。

---

## 10. 技术栈

- **后端**：Python（复用 `wc_value.py` 逻辑）。FastAPI 提供 `/state` JSON + serve 前端 + `/reports` markdown 渲染。
- **存储**：SQLite（盘口历史、价值、watchlist、账本、配置）。
- **前端**：单页 HTML + 原生 JS（或轻量 Alpine/HTMX），每 N 秒拉 `/state` 自动刷新。无重框架。
- **LLM**：Claude API（最新 Opus/Sonnet），**事件驱动**调用（首发出炉 / 分歧拉大），控成本。
- **部署**：docker-compose 一键起；Nginx + 口令（basic auth）；HTTPS。

## 12. 数据流

1. poller 每 1–5min 拉两平台 → 写 SQLite 快照。
2. value-engine 算 value → 更新 `/state`。
3. 新 +EV 点 / 盘口分歧 / 停售临近 → 标记告警。
4. 开赛前 ~1h，lineup-watcher 抓首发 + 新闻 → llm-worker 叠加分析 → 更新关注项/价值点。
5. 前端轮询 `/state` 渲染；用户操作（pin、记账）写回后端。

---

## 13. 范围外（明确不做）

- 不自动下注、不接资金、不代购彩。
- 不承诺 +EV / 盈利；不构成投资建议。
- 不做社交/分享/多用户（单人自用）。
- v1 不做 LLM 概率叠加与构造器（放 v2）。

---

## 14. 里程碑

- **v1**：HK 部署 + poller + value-engine + 价值雷达 + 特别关注 + 账本 + 报告 tab + 停售倒计时。→ 赚小钱最小闭环。
- **v2**：两平台分歧告警 + LLM 首发叠加 + 爽一把构造器 + 追损检测。
- **v3（可选）**：盘口历史图表、回测、每日摘要推送。
