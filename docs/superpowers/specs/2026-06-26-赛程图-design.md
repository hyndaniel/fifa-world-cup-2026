# 赛程图 · 决策面板重做 — 设计文档

- 日期:2026-06-26
- 状态:已通过 brainstorming,待写实现计划
- 范围:重做现有「决策面板」(`#view-decisions`)为可扫赛程 + 常驻右侧详情面板;并修一个决策卡永不过期的日期 bug。

## 1. 背景与问题

现有「决策面板」每场一张完整决策卡(`buildDecisionCard`),竖排铺开。两个痛点:

1. **当天比赛把页面拉得很长**——每张卡默认铺开 v1/v2/价值,场次一多就要长滚动,无法快速"扫盘"。
2. **决策卡永不过期(已诊断的 bug)**——`db.get_decisions()` 取**全部** decisions、零日期过滤,只按 `ko_bj` 升序;写入按 `match_key` upsert、从不删旧卡。后果:已踢完的场卡片一直挂着,次日跑 `/跑今天` 只追加新场、旧卡继续堆积。比看板的"今日比赛"(至少按北京日历日筛、跨午夜自动滚)更严重。

## 2. 目标 / 非目标

**目标**
- 同一视图兼顾两种使用时刻:白天**快速扫盘**(哪场有真 +EV 值得动手)+ 赛前 1h **逐场细看**。
- 通过折叠 + 右侧常驻面板,根治"拉得很长"。
- 修掉决策卡永不过期:过了赛后一段时间自动消失,已开赛的淡化但当晚仍在,跨北京午夜不误删今晚夜场。

**非目标**
- 不新增 tab、不新增接口、不新增数据表。数据口照旧走 `/api/decisions`。
- 不重写决策卡内容——**复用现有 `buildDecisionCard()`** 作为右侧面板/内联展开的内容。
- **报告 tab「全 17:47 / 最上方非最新」**的部署 mtime bug 是独立问题,不在本设计内,单开一条修。

## 3. 设计

### 3.1 布局(前端:`frontend/app.js` + `frontend/style.css`)

最终形态 = **A 折叠行 + C 筛选条 + B 常驻右侧面板**(原型变体 D)。

- **顶部筛选条**:`未结束 / 全部 / 仅绿(真+EV)`,纯前端过滤。**默认选「未结束」**(进页面只看没踢的场,手动切"全部"才显示已结束)。
- **主体 = 单列折叠行**(A):一行一场,内容 `信号灯 · 开球时间 · 主 vs 客 · v1比分 · 最不亏腿 EV 胶囊`。按分组标题(今日未结束 / 已结束)排列;已结束的行淡化(`opacity` 降)、置底。
- **详情 = 常驻右侧面板**(B,非浮层抽屉):点任一行 → 右面板换成那场**整张决策卡**(`buildDecisionCard(d)`)。列表与详情同屏,可边扫边对照。选中行左侧有按 flag 着色的高亮条。
- **响应式**:
  - `≥ 820px`:`grid-template-columns: 1fr 372px`,右侧面板 `position: sticky`。
  - `< 820px`(手机/窄屏):无右栏,点中的行**正下方内联展开**决策卡(退化为 A 行为)。用 `matchMedia('(max-width:820px)')` 在渲染时决定面板挂载位置。

### 3.2 日期处理(后端:`backend/db.py`、`backend/state.py`)

用**「开球时刻 ± 小时」滑窗**替代脆弱的日历日前缀判定:

- 新增纯函数 `decisions_view(decisions, now, decay_h=6)`(在 `state.py`),`/api/decisions` 在 web 层套用它;`db.get_decisions()` **保持不动**(取全部),过滤/标注只在视图层做。它只返回 **`ko >= now − DECAY_H`** 的场(`DECAY_H` 默认 6 小时),按 `ko` 升序。
  - 每条附一个**状态标记** `view_status`:`ko >= now` → `upcoming`;`now > ko >= now − DECAY_H` → `recent`(前端归入"已结束/进行中"组、淡化)。
  - 早于 `now − DECAY_H` 的场**不返回**(自动消失,治堆积)。
- `ko` 解析:`ko_bj` 形如 `"M.D HH:MM"`,按北京时区拼成 datetime;跨年/跨月按当前年补全(沿用 `state.py` 既有的 `_parse_cutoff_dt` 思路)。解析失败的条目保守保留、排末尾。
- **看板「今日比赛」同口径**:`state.py` 的 `matches_today` 也改用同一滑窗判定(替换现有 `today_head` 日历日前缀比对),两处行为一致;顺带修掉看板"已踢完仍显示到午夜"的老问题。

> 为什么用滑窗而非日历日:今晚一批场常含北京 02:00/03:00(次日日期前缀)的夜场,日历日前缀会把同一批夜场拆到两天、或在午夜误删。以"开球时刻"为锚天然连续,不在午夜断裂。

### 3.3 数据流

```
/api/decisions
  → decisions_view(db.get_decisions(), now): 滑窗筛(ko >= now-6h) + 标 view_status(upcoming/recent) + 按 ko 升序
  → 前端筛选条二次过滤(未结束[默认] / 全部 / 仅绿)
  → 渲染: 分组折叠行(今日未结束 / 已结束) + 选中场 → buildDecisionCard 进右面板(或窄屏内联)
```

无新接口、无新表;后端新增 `decisions_view`(web 层套用,`get_decisions` 不动)+ 改 `matches_today` 的判定口径,前端重写 `renderDecisions` 一带的渲染。

## 4. 影响的文件

| 文件 | 改动 |
|---|---|
| `backend/db.py` | `get_decisions()` 加 `now` 入参、滑窗筛选 + status 标注 |
| `backend/state.py` | `matches_today` 改用同一"开球滑窗"判定 |
| `backend/web.py` | `/api/decisions` 传入 `now_bj` 给 `get_decisions` |
| `frontend/app.js` | 重写决策面板渲染:筛选条 + 折叠行分组 + 常驻右侧面板/窄屏内联;复用 `buildDecisionCard` |
| `frontend/style.css` | 折叠行、筛选条、右侧面板、响应式断点 820px、已结束淡化样式 |

## 5. 测试

- **后端单测**(`tests/`):造 `ko` 在 `now` 前/后、`now−6h` 边界两侧、跨北京午夜、解析失败几类 decision,断言 `get_decisions(now)` 的筛选与 status 标注正确;`matches_today` 同口径用例。
- **前端手测**:本地 mock server + Chrome headless 截图,验证:默认进页只见未结束;切"全部"显示已结束(淡化置底);点行右面板换内容;窗口拖窄 < 820px 退化为行内展开;"仅绿"筛选只剩 edge 场。

## 6. 验收标准

1. 进决策面板默认只看**未结束**场;可切"全部 / 仅绿"。
2. 点行 → 桌面右侧常驻面板换成该场决策卡;窄屏行内展开。
3. 赛后超过 6h 的决策卡**不再出现**;已开赛未过 6h 的淡化但仍在。
4. 跨北京午夜不误删当晚夜场;看板"今日比赛"同口径,不再把已踢完的留到午夜。
5. 决策卡内容与现状一致(复用 `buildDecisionCard`,无回归)。
