# 设计文档：一键跑日 + 看板重定心

日期：2026-06-25
状态：已与用户逐块确认，待用户审阅本文档

## 1. 背景与问题

世界杯项目当前有两套并行的东西：

1. **每日预测仪式**——靠在 Claude Code 里"找我聊"驱动：手动唤 agent → WebSearch 盘口 → 综合三份先验 → 手写 `reports/小组赛比分预测.md` → 赛后回填。
2. **价值看板**（已部署 AWS-HK，FastAPI + vanilla JS 前端）——拉足彩 + Polymarket 算价值。

用户反馈的摩擦（多选确认）：**① 每日预测要聊天驱动；② 信息散在多处（预测 md / 看板 / 三份先验 / 复盘各自为政）；③ 价值分析每次要现喊 agent。** 唯独不嫌部署/代理运维麻烦。

用户进一步澄清的两条硬约束：

- **Anthropic 只能本地 macOS 订阅登录，无独立 API key** → 任何 LLM 现推只能在本地 Mac 跑，HK 后端不能直调模型。
- **前端"基本没点开过，觉得功能不好"** → 现有前端 hero 是价值雷达，但足彩长期被 WAF 拦致雷达常空，且用户每天真正关心的预测埋在报告 tab 里——产品中心与用法错位。

## 2. 目标 / 非目标

**目标：**
- 用**一个本地动作**（一个 skill，如 `/跑今天`）替代每日的长聊：自动算出该预测哪些场 → 跑 v1/v2/价值 → 写回 → 渲染到看板。
- 把散落的预测/概率/价值/情报**收口到看板的"每场决策卡"**，打开就是当天要的东西。
- 前端**重定心 + 复用现有技术栈**（vanilla JS + 现有 CSS，不上框架）。

**非目标（明确不做）：**
- ❌ HK 后端直调 Claude API（约束否决）。
- ❌ 任务队列 / 本地 worker / Mac-on-phone 桥接（用户选"人在 Mac 前一个动作搞定"，不要求手机触发预测）。
- ❌ 前端框架重写（React/Vue 等）。
- ❌ 改动部署/代理运维。
- ❌ 自动下注、碰资金。

## 3. 架构决策（锁定）

| 决策 | 选择 | 理由 |
|---|---|---|
| LLM 跑在哪 | 本地 Mac | Anthropic 仅本地订阅登录，无 key 可下放 HK |
| 触发形态 | 本地一个 skill（人在 Mac 前） | 用户选"一个动作搞定"，不要求手机触发预测 |
| 价值侧 | 确定性 Python，留在 HK，看板加刷新按钮 | 无需 LLM；手机查看不依赖 Mac 开机 |
| 前端 | 重定心 + 复用 vanilla JS/CSS | 用户确认诊断"摆错中心"，不上框架 |
| v1 与 v2 | 独立 pass，预测时刻互不可见 | 护三方 Brier 对照红线 |

## 4. 组件设计

### 4.1 组件 A：`/跑今天` 编排 skill（本地）

一个 Claude Code skill，用户敲一次即串完整套，替代长聊：

1. 读 `reports/小组赛比分预测.md` 看进度 → 按 ET 当下时刻算出"今天/明天该预测哪几场"（时区纪律见 wc2026-prediction-workflow 第 10 条）。
2. 拉最新盘口：足彩走本地采集器（已有 launchd 常驻），Poly 去水（刷新防陈旧假黄档）。
3. **并行派三条独立 pass（关键：互不共享上下文）：**
   - **v1** = `football-match-predictor`：比分 + 出线，落 `v1_predictions` + 写回 `小组赛比分预测.md`。
   - **v2** = `wc-forecaster-v2`：**只喂** `baseline_market(...)` + `match_fact_card(...)`（中立事实卡，经 `tools/save_intel.py` 从 deep-search 情报进库的确证事实）→ `record_v2_prediction`。**绝不把 v1 的任何输出传入 v2 的上下文。**
   - **价值** = `odds-value-analyst`：green/yellow + 最不亏腿。
4. 跑 `tools/v2_report.py` → 渲染 `reports/预测v2.md`（v1/v2/市场 Brier 跑分卡）。
5. **写回看板**（见 4.3）。
6. 给用户一段摘要供扫读；要改的继续聊（保留聊天通道做"改预测"）。

**诚实边界（写进 skill）：** 它做掉 routine 的约 80%；**赛前 ~1h 首发微调**（边际收益最高的窗口，见 workflow 第 11 条）仍需用户在临场再触发一次或由 skill 提示——首发名单是临场出炉的，不假装全自动。

**v1/v2 独立性的实现保证：** 在 skill 编排里，v1 与 v2 作为**各自独立的 subagent 调用**，v2 的 prompt 只注入市场基线与事实卡，绝不注入 v1 的 probs/score。展示层并排显示不违反此红线（红线管"预测时刻别互看"，不管"查看时并排"）。

### 4.2 组件 B：看板重定心（前端）

复用 vanilla JS + 现有 832 行 CSS，**改信息架构与 hero**：

- **新 hero = 每场一张「决策卡」**（按开球时间排序），每卡含：
  - 对阵 + 国旗 + 开球 ET（附北京时间）
  - **v1 比分预测**（+ 一句出线诉求/依据）
  - **v2 概率**（胜/平/负 %）+ **靠谱度（稳/中/乱）** + **剧本标签**
  - **价值结论**（🟢真+EV / 🟡接近公允 / 该场别碰 + 最不亏腿）
  - 展开 → 完整盘口明细（让球/大小球、足彩隐含 vs Poly 去水）
- **次级 tab**：价值雷达（保留，加"重抓+刷新"按钮，连带刷 Poly）、A/B 账本、特别关注、报告 md。
- 移动优先（沿用现有 viewport/dark 设置）。

### 4.3 组件 C：预测写回看板的通道

把本地跑出的预测送达 HK 看板，让决策卡有数据。复用现有模式（足彩已用 `POST /api/ingest/zucai` 从 Mac 推 HK）：

- 新增 ingest 端点（如 `POST /api/ingest/predictions`），接收 v1 比分 + v2 概率/靠谱度/剧本 + 价值结论，存入看板 DB。
- 新增读取端点（如 `GET /api/decisions`），前端决策卡据此渲染。
- v2 概率本就落在 `v2_predictions` 表；v1 在 `v1_predictions` + markdown。写回通道负责把这些汇成决策卡所需结构。

## 5. 数据流

```
[本地 Mac]  /跑今天 skill
   ├─ 读 小组赛比分预测.md 进度 + ET 判定待预测场次
   ├─ 取盘口(足彩本地采集 / Poly 去水)
   ├─ 并行：v1(比分,出线) ‖ v2(基线+事实卡→概率) ‖ 价值(EV/最不亏腿)   ← v1⊥v2
   ├─ v2_report.py → 预测v2.md (Brier 跑分卡)
   ├─ 写回 小组赛比分预测.md
   └─ POST /api/ingest/predictions ─────────────┐
                                                  ▼
[AWS-HK]  看板后端  ── GET /api/decisions ──►  前端「每场决策卡」(手机可看)
   └─ 价值侧: 确定性重抓+重算 (刷新按钮, 不依赖 Mac)
```

## 6. 复用 / 新建 / 不建

- ♻️ **复用**：三个 agent 的推理栈、`value.py`/`devig.py`/`baseline.py`、本地 launchd 足彩采集、`/api/reports`、`v2_predict.py`/`v2_report.py`/`v1_log.py`、现有前端骨架与 CSS。
- 🆕 **新建**：① `/跑今天` 编排 skill（薄编排层，保证 v1⊥v2）；② 前端决策卡视图 + 信息架构重排 + 价值刷新按钮；③ 写回通道（`/api/ingest/predictions` + `/api/decisions`）。
- 🚫 **不建**：HK 调模型、任务队列、worker、前端框架、新部署链路。

## 7. 测试

- 写回通道：后端端点单测（沿用现有 fixture 级 pytest 风格，`test_web.py` 扩展）。
- v1⊥v2 独立性：编排层需有检查/约定确保 v2 上下文不含 v1 输出（设计约束，实现期落为显式校验或代码结构隔离）。
- 前端决策卡：以 mock `/api/decisions` 响应渲染，验证空数据（无盘口/无预测）降级显示不崩。
- 价值刷新：复用现有 value_radar 测试路径。

## 8. 开放问题

- `/跑今天` 是做成 Claude Code skill（`/<name>` 调用）还是本地 CLI 脚本——倾向 skill（贴合用户日常在 Claude Code 内操作的习惯），实现计划阶段定稿。
- 决策卡的"价值结论"在前端是直接展示后端算好的，还是展示 skill 跑出的快照——倾向后端确定性算 + skill 快照择新，实现计划阶段定。
