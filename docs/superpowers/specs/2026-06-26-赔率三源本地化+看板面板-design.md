# 赔率三源本地化 + 看板赔率面板 — 设计

日期:2026-06-26
状态:已批准(approach A + 合并为单一 launchd 任务)

## 目标

1. **简化流程**:把现在"竞彩(collect_zucai launchd)+ 欧盘(手动 odds_watch --consensus)+ Poly(经 HK remote-agent 手动)"三套刷新,收敛成**本地一个定时任务**。
2. **看板可见**:打开 HK 看板,在每张决策卡里看到**三源现价 + 相对上次的变化 + 竞彩vs欧盘分歧**。

## 关键前提(实测)

- **Poly 经本地 7897 代理可直抓**(日本节点;`gamma-api` 端到端跑通,18 场即时去水)。`poly_fetch_hk` 用 urllib,自动读 `https_proxy` 环境变量 → 仅需设环境变量即本地化,**不再依赖 aws-hk / remote-agent MCP**。
- **抓取必须在本地发生**:Poly 走本地代理、竞彩 sporttery 对 HK 数据中心 IP 有 WAF 拦截 → HK 自身抓不了。故看板新鲜度只能靠"本地抓完推到 HK"。
- **sporttery ≠ 500 竞彩官方**:500 是滞后镜像(实测 5 场 3 同 2 异,最大主胜 7.1 vs 6.85),且 500 `ouzhi` 页只给竞彩 1X2。→ **竞彩源保持 sporttery**(官方价、含 had/hhad/ttg 三盘、本地直连)。封盘后竞彩线冻结,sporttery 封盘前最后快照即最终价,无新鲜度损失;看板的 24h 活信号由欧盘提供。
- 三源在本地缓存的 key 不一致:竞彩=`zucai_num`(如 `周四055`)、欧盘=`500-<fid>`、Poly=label。需对齐到 `zucai_num`(= 看板 `match_key`)。

## 架构(approach A:本地算好,HK 只存只显)

本地 **一个 launchd 任务** `com.wc.refresh-all`(默认每 5 分钟,住宅 IP)跑 `tools/refresh_all.py`:

```
refresh_all (每5min, 住宅IP)
 ├─ odds_watch --consensus            → 本地缓存(竞彩 had/hhad/ttg + 欧盘 1X2 共识)
 ├─ poly_fetch fetch(https_proxy=7897) → 本地缓存(Poly 1X2 去水)
 ├─ 读缓存: 三源 latest+prev → 按队名对齐到 zucai_num, 算 Δ变化 + 竞彩vs欧盘分歧
 ├─ POST /api/ingest/zucai            (竞彩 raw 信封, 喂决策流; 取代 collect_zucai)
 └─ POST /api/ingest/odds             (对齐好的赔率面板 payload[])
HK 看板:
 ├─ /api/ingest/odds → db.save_odds(match_key, payload)   (按 match_key upsert, 替换语义)
 ├─ /api/decisions → decisions_view 里 join db.get_odds(match_key) 进每卡 odds 字段
 └─ 前端 app.js 在决策卡内渲染可折叠赔率面板
```

**少一套体现**:本地从三套刷新 → 收敛成 `refresh_all` 单一 launchd;取代独立的 `collect_zucai`(决策流的竞彩 raw 由 refresh_all 一并推)。

## 赔率面板 payload 形状(本地算好)

每场一项:
```json
{
  "match_key": "周四055",
  "label": "土耳其 vs 美国",
  "ko": "2026-06-26 10:00",
  "fetched_at": "2026-06-26T10:00:00+08:00",
  "sources": {
    "zucai":     {"had": {"h":2.7,"d":3.4,"a":1.95}, "hhad": {"line":-1,"h":..,"d":..,"a":..}, "ttg": {"0":..,"1":..}, "delta": {"had": {"h":"▲","d":"-","a":"▼"}}, "stale": false},
    "consensus": {"devig": {"h":34.5,"d":27.4,"a":38.1}, "euro": {"h":2.7,"d":3.4,"a":2.45}, "n_books": 47, "delta": {...}, "stale": false},
    "poly":      {"devig": {"h":..,"d":..,"a":..}, "delta": {...}, "stale": false}
  },
  "divergence": {"h": +2.1, "d": -1.0, "a": -1.1}   // 竞彩去水 − 欧盘去水 (pp), 标分歧
}
```
- `delta`:相对**上一快照**的方向(▲升/▼降/-平),由本地读缓存 latest vs prev 算。
- `divergence`:竞彩去水% − 欧盘共识去水%;任一选项 `|divergence| ≥ 3pp` 时该卡打分歧徽标(阈值定为 3pp)。
- 某源缺/超时 → 该源 `stale:true` 或省略该 source,面板显"—/陈旧",其余照常。

## 前端面板(完整范围)

决策卡内可折叠区,内容:
- **三源 × 主/平/负**:竞彩现价、欧盘共识去水%(或现价)、Poly 去水%;每格带 ▲▼ 变化。
- **竞彩让球(hhad)**:让球线 + 主/客。
- **竞彩总进球(ttg)**:各档赔率(精简显示 0–4+ 或折行)。
- **分歧徽标**:竞彩明显偏离欧盘共识处高亮。
- **抓取时间** `fetched_at`。
- 视觉:轻量小表 + 变化箭头配色(▲红升水/▼绿降水,沿用项目水位语义),默认折叠。

## 错误处理(best-effort,缺源明说不假算)

- 某源抓失败/超时 → 该源 `stale`,面板显"—/陈旧",其余源照常推。沿用项目"缺源明说、不拿陈旧硬算"原则(见记忆 Stale Poly = false yellow)。
- 对齐失败(队名配不上 `zucai_num`)→ 跳过该场赔率面板(决策卡仍在),日志记 unmatched 队名。
- 推 HK 失败(端点 404/网络)→ 本地缓存已写,下周期重推;launchd 不中断单次失败。
- `/api/ingest/odds` 未部署前,推送 404 → 本地容错记日志,不报错。

## 测试

- `poly_fetch_hk` 代理抓取:离线 fixture 解析(已有 `tests/fixtures/poly_*.json`)+ 一次真实代理连通自检 `--selftest`(可跳过 if 无代理)。
- 对齐逻辑:构造三源缓存(竞彩 `周X0NN`、欧盘 `500-fid`、Poly label)→ 断言 join 到正确 `zucai_num` + Δ/分歧计算。
- `/api/ingest/odds` + `save_odds`/`get_odds`:沿用 `tests/test_decisions_view.py` 模式,断言 upsert 替换语义 + decisions join 出 `odds` 字段。
- `refresh_all --once --dry-run`:冒烟(抓取+对齐,不推 HK)。

## 部署

- HK **一次性 deploy**(新端点 + 前端面板,走 `fifa-deploy` skill 的 docker build)。之后赔率纯靠 ingest,数据更新不再 redeploy。
- launchd:新增 `com.wc.refresh-all.plist`,**停用/卸载旧的 `com.wc.collect-zucai`**。plist 设 `WC_INGEST_URL/PW`、`WC_HTTPS_PROXY`、间隔。
- **go-live 两步(deploy + launchd 切换)触及生产/实盘机,留作显式确认步骤,不在 AFK 自动执行范围**。代码实现 + 测试 + 合并 main 为 AFK 交付范围。

## 非目标(YAGNI)

- HK 不留赔率历史、不画走势图(approach A 只存最新一份)。
- 不用 500 替换 sporttery 竞彩。
- 不做赔率告警/推送(odds_watch 的 ▲▼ 日志已够盯盘)。
