# 彩票入库 —— 购彩票据归档命名

把用户每天存放新票据照片的暂存文件夹,逐张规范成统一命名后搬进
`/Users/heyining/Downloads/购彩记录/按人整理/`(下称"归档目录"),**归档完成后自动派 wc-bet
子代理补录台账**——本 skill 自己只做文件改名+搬运+触发记账,**不直接编辑
`data/bet_ledger.json`**;台账的字段口径(picks 拆解、查重判断等)
一律交给 wc-bet 按它自己维护的规则处理,避免两处各写一套导致口径漂移或互相覆盖
(历史上已出现过 074-082/077-084 两张票被误判成同一张互相覆盖的事故)。

## 输入 / 输出约定

- **输入**:用户给一个暂存目录路径,里面是本次要入库的照片,文件名规则固定为
  **"人名缩写+编号"**(例如 `HYN1.jpg`、`LYH3.JPG`)——**人名已经在文件名里,不用开图猜、不用问用户**。
  已知人名缩写集合(会随家庭成员增减,不强行校验,原样按前缀取用即可):`HYN`(用户本人)、
  `LYH`、`ZFW`、`LYZ`、`YBB`。
- **输出**:归档目录下的规范文件名——

  ```
  人名_期号_类型(腿号).扩展名
  ```

  - **人名**:直接取原始文件名的人名缩写前缀,原样用(不做归一化/翻译)。
  - **期号**:票面上打印的实际期号数字,**不是**下注/拍照日期。竞彩票据顶部"体彩/竞彩"
    logo 下方有一行"第XXXXXXX期",只取数字部分(去掉"第""期"两个字)。**必须开图核对**,
    不能从文件名/日期推断——同一天下的多张票、甚至同一人同一天不同玩法的票,期号完全可能
    不同(已有实例:HYN 06-19下的票期号却是 2606211;06-23下的票期号是 2606231)。
  - **腿号**:该票覆盖的场次编号或范围,取自票面"第1场…第N场"列表的场次三位数编号
    (如 077、084),写成 `077-084`;只有一场就写单个编号 `073`。**只留数字本身**,
    球队名、"夜场""跨期""期"等描述性文字一律去掉。
  - **类型**:玩法+关系结构的简短描述,格式参照归档目录里已有文件(见下方"类型段写法参考")。
    这段允许保留描述性后缀(例如"中奖"表示这张票中了),因为它描述的是这张票本身而不是
    腿号——**只清理腿号括号里的内容,不清理类型段**。
  - 扩展名大小写跟原图保持一致。

## 类型段写法参考(照抄归档目录已有命名的风格,不要发明新格式)

竞彩票面上能读到的关键信息 → 类型段怎么写,举例:

| 票面信息 | 类型段写法示例 |
|---|---|
| 单关(只1场) | `胜平负单关` / `半全场单关` / `比分单关` |
| N场N串1(单式,只1种组合) | `让球4串1` / `胜平负3串1` / `混合8串1` |
| N场2-M关复式(多种串关组合叠加) | `让球6场4关复式` / `混合5场3关复式` |
| N场2-M关**全组合**(票面写"共X注"覆盖2到N关所有组合) | `混合8场2-8关复式` 或标注注数如
  `混合2-8关全组合(247注)`(参照归档目录里 `HYN_2607021_混合8场2-8关复式(077-084).jpg` 这类) |
| 单式但下了多倍/多张相同实票 | 在类型段末尾加 `×N倍` 或"两张"/"3张"等,如 `胜平负3串1×2倍`、
  `单场固定×50倍3张` |
| 半全场玩法 | `半全场单关` / `半全场3串1` |
| 比分玩法 | `比分单关` / `比分3串1复式` |
| 北京单场(北单,非竞彩) | 票种和期号体系跟竞彩不同,期号可能只有5位数(如 `26068`),
  照票面实读,不要套用竞彩7位数假设;类型段照常写玩法,如 `北单3关+4串1两张` |

拿不准某张票该归哪类写法时,优先照抄归档目录里同玩法已有文件的措辞,保持风格一致;
实在是新玩法结构,选一个最贴票面描述的简短说法即可,不必强求完全吻合旧例。

## 执行步骤

1. **列出暂存目录里的文件**,每个文件名解析出人名前缀(去掉后面的编号/扩展名)。
2. **逐张用 Read 工具把图片当图看**,从票面读出:
   - 期号数字("第XXXXXXX期"一行)
   - 起止场次编号("第1场…第N场"列表的首尾三位数场次号)
   - 玩法结构(单关/串1/复式/全组合/倍数等,按上表写成类型段)
3. **拼出目标文件名**:`人名_期号_类型(腿号).扩展名`。
4. **搬运前查重**:归档目录里若已有**同人名+同期号+同腿号**的文件(哪怕类型段措辞写法不同),
   视为疑似重复入库——类型段允许多种合理写法,不能只靠"目标文件名完全相同"来判重,否则
   同一张票用词稍有出入就会绕过保护、重复归档。查到疑似重复,**停下来报告,不搬运**,交人工确认
   是否真是同一张票。
5. **搬运**:确认非重复后,用 `cp` 从暂存目录**复制**到归档目录 `/Users/heyining/Downloads/购彩记录/按人整理/`——
   **用 `cp` 不用 `mv`,原图留在暂存目录做底**:票多了,归档名里万一读错期号/场次,能回原图逐张比对复查。
   原图不要动、不要删;暂存目录何时清理由用户自己决定,本 skill 不代删。
   - 若目标文件名(完全相同)在归档目录已存在,同样**停下来报告冲突,不覆盖**。
   - 腿号括号固定放在文件名最末尾、用全角括号 `(` `)`;类型段自身若也带括号(如"全组合(247注)"
     这类注数说明),那对括号写在类型段内部、不要和腿号的括号相邻到让人分不清哪对是腿号
     (例如宁可写 `混合2-8关全组合247注(077-084).jpg`,把注数并进类型段文字里,不单独开括号)。
6. **自动记账**:本轮所有票据搬运完成后(不必等用户开口),主动用 Agent 工具派一个
   `wc-bet` 子代理,把**本次成功归档的每张票**的归档后文件**绝对路径**(子代理没有
   与主会话相同的 cwd 保证,必须给绝对路径)交给它,请它:
   - 逐张开图读出实际下注选项(picks/赔率),按它自己维护的台账口径,给每张票在
     `data/bet_ledger.json` 里补一条 `settled: false / pnl: null` 的未结算记录;
   - `who` 字段直接写人名缩写本身(HYN 本人也写 `HYN`,不再用"你"这类别名;其余人名前缀照抄);
   - 若发现同 serial(期号)+ who + 腿号已有记录,自行判断是否重复,重复则跳过并说明,
     不要写出两条指向同一张实体票的记录。
   - 疑似重复/冲突/图片读不清的票据(如本文档"边界"里提到的需要人工确认场景),
     wc-bet 应停下报告而不是瞎编。
   派出后等待 wc-bet 的结果,记录它写入/跳过了哪几张。

   > **🔴 硬约束——记账必须落到 main 的规范台账,严禁停在未合并的 worktree:**
   > wc-bet 写的 `data/bet_ledger.json` **必须是 main 分支跟踪、也是 HK 看板读取的那一份规范台账**,
   > **绝不能只在某个 worktree 副本里改完、把 commit 留在未合并的分支上就收工**——那会造成"照片入了库、
   > 台账却进不了 main",主会话和看板都看不到(2026-07-02 实际事故:083-085/087 两批共 11 张票的记账
   > commit 落进 3 个未合并的 worktree 分支,main 台账停在 63 张,直到人工发现才补回)。落地做法二选一:
   > - **首选**:记账直接改**主工作区(main 检出)**的 `data/bet_ledger.json`——即使本次入库运行在后台 job 的
   >   隔离 worktree 里,派 wc-bet 时也要把**主工作区仓库根的绝对路径**交给它(不是当前 worktree 那份副本)。
   > - 或:写完后**立即**把台账改动落到 main——走 `数据快推` skill 直接 POST 到看板(看板读的就是 main 那份),
   >   或把这条记账 commit 合并/推送进 main;**不允许**只推到 worktree 分支就结束。
   >
   > 收尾自检(见第 7 步)必须确认:改动已出现在 **main 跟踪的 `data/bet_ledger.json`** 或已同步到看板,
   > 而**不是**只存在于某个 worktree 副本/分支。
7. **收尾**:给用户一张"暂存文件名 → 归档后文件名(含读到的期号)"对照表,方便核对;
   附上第 6 步 wc-bet 的记账结果(成功记入 X 张 / 跳过 Y 张及原因)。
   **落地自检(硬性)**:确认这批记账已写进 **main 跟踪的 `data/bet_ledger.json`**(或已通过 `数据快推`
   同步到看板),**不是**只留在某个 worktree 副本/未合并分支里——见第 6 步🔴硬约束。

## 处理量大时的并行策略

单次入库的照片可能有十几二十张。人手多、图片多时,按**人名分组**派几个子代理并行处理
(每组几张到十几张):
- 每个子代理只认领自己那组的文件,不碰其他组的文件,避免相互踩踏。
- 给每个子代理:该组的完整原始文件清单(路径)、本文件里的完整改名规则(期号读取方式、
  腿号清理规则、类型段写法参考、查重规则)、归档目标目录。
- 子代理完成后各自回报"暂存名→归档名(含读到的期号)"对照表,由主会话汇总成一张总表
  交给用户核对,并汇总提示是否有子代理报告了疑似重复/冲突/读不清的情况。
- 归档分组子代理**只管改名搬运,不派 wc-bet**;等所有分组都汇总完,由主会话在收尾阶段
  统一派**一次** wc-bet(带上本轮全部新归档文件路径),避免多个分组各自触发记账、
  互相踩踏或重复记同一批票。

## 边界

- 不生成/不猜测任何票面上没写的信息(期号、场次、赔率一律照票面读)。
- 本 skill(主会话这一层)自己不直接编辑 `data/bet_ledger.json`——记账动作委托给
  wc-bet 子代理执行,由它按自己的台账口径写入,不在本 skill 里另拼一套 picks/字段逻辑。
- **🔴 记账落 main、不进 worktree**:wc-bet 写台账必须落到 main 的规范 `data/bet_ledger.json`
  (主工作区 / 看板读的那份),不允许只改 worktree 副本、把 commit 留在未合并分支——详见第 6 步硬约束与自检。
- 不删除、不移动原始照片:用 `cp` **复制**进归档目录、**原图留在暂存目录做底**,方便日后读错期号/场次时回原图复查。暂存目录何时清理由用户决定,本 skill 不代删、不 `mv`。
- 遇到人名前缀不在已知集合、或图片模糊读不出期号/场次的情况,停下来问用户,不要瞎编。
- 若用户明确要求"只归档、先别记账"之类的临时指示,以用户当次指示为准跳过第 6 步。
