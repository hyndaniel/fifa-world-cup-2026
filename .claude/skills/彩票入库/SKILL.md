# 彩票入库 —— 购彩票据归档命名

把用户每天存放新票据照片的暂存文件夹,逐张规范成统一命名后搬进
`/Users/heyining/Downloads/购彩记录/按人整理/`(下称"归档目录")。**只做文件改名+搬运,不碰
`data/bet_ledger.json`**——把赛果/赔率/盈亏记进台账是 wc-bet 的活,归档完可以另外喊它补录。

## 输入 / 输出约定

- **输入**:用户给一个暂存目录路径,里面是本次要入库的照片,文件名规则固定为
  **"人名缩写+编号"**(例如 `HYN1.jpg`、`LYH3.JPG`)——**人名已经在文件名里,不用开图猜、不用问用户**。
  已知人名缩写集合(会随家庭成员增减,不强行校验,原样按前缀取用即可):`HYN`(用户本人)、
  `LYH`、`ZFW`、`LYZ`、`YBB`。
- **输出**:归档目录下的规范文件名——

  ```
  人名_期号_类型(腿号).扩展名
  ```

  - **人名**:直接取原始文件名的人名缩写前缀,原样用(不做归一化/翻译)。
  - **期号**:票面上打印的实际期号数字,**不是**下注/拍照日期。竞彩票据顶部"体彩/竞彩"
    logo 下方有一行"第XXXXXXX期",只取数字部分(去掉"第""期"两个字)。**必须开图核对**,
    不能从文件名/日期推断——同一天下的多张票、甚至同一人同一天不同玩法的票,期号完全可能
    不同(已有实例:HYN 06-19下的票期号却是 2606211;06-23下的票期号是 2606231)。
  - **腿号**:该票覆盖的场次编号或范围,取自票面"第1场…第N场"列表的场次三位数编号
    (如 077、084),写成 `077-084`;只有一场就写单个编号 `073`。**只留数字本身**,
    球队名、"夜场""跨期""期"等描述性文字一律去掉。
  - **类型**:玩法+关系结构的简短描述,格式参照归档目录里已有文件(见下方"类型段写法参考")。
    这段允许保留描述性后缀(例如"中奖"表示这张票中了),因为它描述的是这张票本身而不是
    腿号——**只清理腿号括号里的内容,不清理类型段**。
  - 扩展名大小写跟原图保持一致。

## 类型段写法参考(照抄归档目录已有命名的风格,不要发明新格式)

竞彩票面上能读到的关键信息 → 类型段怎么写,举例:

| 票面信息 | 类型段写法示例 |
|---|---|
| 单关(只1场) | `胜平负单关` / `半全场单关` / `比分单关` |
| N场N串1(单式,只1种组合) | `让球4串1` / `胜平负3串1` / `混合8串1` |
| N场2-M关复式(多种串关组合叠加) | `让球6场4关复式` / `混合5场3关复式` |
| N场2-M关**全组合**(票面写"共X注"覆盖2到N关所有组合) | `混合8场2-8关复式` 或标注注数如
  `混合2-8关全组合(247注)`(参照归档目录里 `HYN_2607021_混合8场2-8关复式(077-084).jpg` 这类) |
| 单式但下了多倍/多张相同实票 | 在类型段末尾加 `×N倍` 或"两张"/"3张"等,如 `胜平负3串1×2倍`、
  `单场固定×50倍3张` |
| 半全场玩法 | `半全场单关` / `半全场3串1` |
| 比分玩法 | `比分单关` / `比分3串1复式` |
| 北京单场(北单,非竞彩) | 票种和期号体系跟竞彩不同,期号可能只有5位数(如 `26068`),
  照票面实读,不要套用竞彩7位数假设;类型段照常写玩法,如 `北单3关+4串1两张` |

拿不准某张票该归哪类写法时,优先照抄归档目录里同玩法已有文件的措辞,保持风格一致;
实在是新玩法结构,选一个最贴票面描述的简短说法即可,不必强求完全吻合旧例。

## 执行步骤

1. **列出暂存目录里的文件**,每个文件名解析出人名前缀(去掉后面的编号/扩展名)。
2. **逐张用 Read 工具把图片当图看**,从票面读出:
   - 期号数字("第XXXXXXX期"一行)
   - 起止场次编号("第1场…第N场"列表的首尾三位数场次号)
   - 玩法结构(单关/串1/复式/全组合/倍数等,按上表写成类型段)
3. **拼出目标文件名**:`人名_期号_类型(腿号).扩展名`。
4. **搬运前查重**:归档目录里若已有**同人名+同期号+同腿号**的文件(哪怕类型段措辞写法不同),
   视为疑似重复入库——类型段允许多种合理写法,不能只靠"目标文件名完全相同"来判重,否则
   同一张票用词稍有出入就会绕过保护、重复归档。查到疑似重复,**停下来报告,不搬运**,交人工确认
   是否真是同一张票。
5. **搬运**:确认非重复后,`mv` 从暂存目录移到归档目录 `/Users/heyining/Downloads/购彩记录/按人整理/`。
   - 若目标文件名(完全相同)在归档目录已存在,同样**停下来报告冲突,不覆盖**。
   - 腿号括号固定放在文件名最末尾、用全角括号 `(` `)`;类型段自身若也带括号(如"全组合(247注)"
     这类注数说明),那对括号写在类型段内部、不要和腿号的括号相邻到让人分不清哪对是腿号
     (例如宁可写 `混合2-8关全组合247注(077-084).jpg`,把注数并进类型段文字里,不单独开括号)。
6. **收尾**:给用户一张"暂存文件名 → 归档后文件名(含读到的期号)"对照表,方便核对;
   同时提醒——**这只是文件归档,票据的赔率/赛果/盈亏还没进台账**,若要记账请另外喊 wc-bet。

## 处理量大时的并行策略

单次入库的照片可能有十几二十张。人手多、图片多时,按**人名分组**派几个子代理并行处理
(每组几张到十几张):
- 每个子代理只认领自己那组的文件,不碰其他组的文件,避免相互踩踏。
- 给每个子代理:该组的完整原始文件清单(路径)、本文件里的完整改名规则(期号读取方式、
  腿号清理规则、类型段写法参考、查重规则)、归档目标目录。
- 子代理完成后各自回报"暂存名→归档名(含读到的期号)"对照表,由主会话汇总成一张总表
  交给用户核对,并汇总提示是否有子代理报告了疑似重复/冲突/读不清的情况。

## 边界

- 不生成/不猜测任何票面上没写的信息(期号、场次、赔率一律照票面读)。
- 不触碰 `data/bet_ledger.json` 或任何台账文件——归档和记账是两件事,分开做。
- 不删除原始照片(用 `mv` 移动,不是复制后残留副本,也不是删除后不归档)。
- 遇到人名前缀不在已知集合、或图片模糊读不出期号/场次的情况,停下来问用户,不要瞎编。
